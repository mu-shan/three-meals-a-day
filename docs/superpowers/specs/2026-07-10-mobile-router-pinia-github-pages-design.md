# 移动端架构与 GitHub Pages 部署设计

## 目标

将现有单路由 Vue 应用升级为面向移动端的双路由家庭菜单应用。使用 Pinia 收口菜单和口味偏好状态，使用 Vue Router 拆分今日菜单与口味偏好页面，增加本地数据导出与导入恢复，并通过 GitHub Actions 自动部署到免费的 GitHub Pages。

线上仓库确定为公开仓库 `mu-shan/three-meals-a-day`。每台设备独立使用浏览器本地数据，不增加账号、后端或设备间同步。

## 产品边界

本次包含：

- 今日菜单与口味偏好双路由；
- 菜单和偏好 Pinia 状态管理；
- 移动端专属布局；
- 本地数据 JSON 导出与导入恢复；
- Git 初始化、公开 GitHub 仓库和 Pages 自动部署；
- 现有菜单规则、喜欢加权和不喜欢过滤能力的无损迁移。

本次不包含：

- 用户登录与家庭账号；
- 云端数据库或多设备同步；
- 历史菜单功能；
- 独立采购清单路由；
- PWA 安装、推送通知或离线缓存；
- 自定义域名。

## 路由结构

应用继续是 SPA，但从单路由页面升级为多路由应用。使用 `createWebHashHistory()`，确保 GitHub Pages 上直接刷新和复制链接不依赖服务端回退。

- `/#/`：今日菜单页面，包含家庭顶部区、早餐、午餐、晚餐和采购清单；
- `/#/preferences`：口味偏好页面，包含喜欢、不喜欢与数据备份管理。

Hash 模式会让线上偏好地址形如 `https://mu-shan.github.io/three-meals-a-day/#/preferences`。相比 Web History，它牺牲少量 URL 美观性，换取 GitHub Pages 上更稳定的路由行为。

路由切换后滚动到页面顶部。未知路由重定向到首页。

## 应用外壳

`App.vue` 只负责：

- 全局移动端画布；
- 路由出口；
- 响应式导航；
- 全局轻量提示。

业务状态、随机动作、本地存储和页面内容不再放在 `App.vue`。

桌面浏览器打开时，应用显示为居中的移动端画布，最大宽度约 `560px`。画布之外保留当前暖黄色纹理背景，页面内容始终保持移动端单列布局，不再维护三列桌面菜单。

所有尺寸统一使用底部固定导航栏，包含“今日菜单”和“口味偏好”两个入口。导航栏考虑 `safe-area-inset-bottom`，内容区增加等量底部间距。电脑端沿用相同信息结构，不扩展为桌面站点。

## 页面设计

### 今日菜单页

首页保留现有家庭氛围和功能：

- 日期、标题与一键开饭；
- 早餐、午餐、晚餐单列卡片；
- 每餐换一桌；
- 每道菜换一道、喜欢和不喜欢；
- 今日采购清单。

原“口味偏好”入口改为路由链接，继续展示偏好总数。点击菜品喜欢或不喜欢后，Pinia 状态立即更新；不喜欢仍需即时换成同类型菜品。

### 口味偏好页

偏好页替代现有侧边抽屉，使用完整页面呈现“家庭口味簿”：

- 页面顶部显示喜欢数量、不喜欢数量和总记录数；
- “喜欢的菜 / 不喜欢的菜”标签切换；
- 菜品使用单列触控卡片展示缩略图、名称、食材和自动归属；
- 喜欢列表提供“取消喜欢”；
- 不喜欢列表提供“重新加入”；
- 空状态引导用户回首页为菜品做标记；
- 页面底部提供数据备份与恢复。

原 `DishPreferenceDrawer` 删除，不保留抽屉与独立页面两套入口。

## Pinia 状态设计

### `preferences` store

负责：

- 恢复与保存喜欢、不喜欢菜品 ID；
- 维护喜欢与不喜欢互斥；
- 提供固定两倍权重所需的只读集合；
- 提供喜欢菜品、不喜欢菜品和数量的派生状态；
- 取消喜欢、标记不喜欢和重新加入；
- 接收导入的偏好数据并一次性替换状态。

store 复用 `dishPreferenceStorage`，不直接包含 JSON 解析细节。

### `menu` store

负责：

- 恢复当天菜单或首次生成；
- 生成全部菜单；
- 重抽单餐；
- 替换单道菜；
- 标记不喜欢后先验证候选，再即时换菜；
- 保存当天菜单；
- 提供采购清单派生状态；
- 维护换菜动画状态和全局反馈；
- 导入同日菜单或在跨日导入后生成当天菜单。

菜单 store 在动作执行时读取偏好 store 的规则，但随机算法继续保留在 `domain/menuGenerator.ts`。领域层不依赖 Pinia、Vue Router 或浏览器 API。

### 状态生命周期

Pinia 在应用挂载前创建。两个路由共享同一 store 实例，因此从首页进入偏好页再返回时，菜单、喜欢状态和动画状态不会重新初始化。

保留当前 `localStorage` 键和数据格式，已有用户升级后可以继续读取原菜单与偏好。清洗后的无效菜品 ID 仍会回写存储。

## 本地数据备份

新增独立备份模块，负责从已校验的应用状态创建备份，以及解析用户选择的 JSON 文件。组件和 store 不直接实现底层文件格式校验。

导出文件包含：

```json
{
  "version": 1,
  "exportedAt": "2026-07-10T08:00:00.000Z",
  "menu": {},
  "preferences": {
    "likedIds": [],
    "dislikedIds": []
  }
}
```

具体规则：

- `version` 固定为 `1`，为未来历史菜单迁移预留版本；
- `exportedAt` 使用 ISO 时间；
- `menu` 保存当前设备的当天菜单；
- `preferences` 保存喜欢和不喜欢记录；
- 导出文件名包含本地日期，例如 `three-meals-a-day-backup-2026-07-10.json`。

导入分为解析、预览、确认、应用四步：

1. 用户选择 `.json` 文件；
2. 校验 JSON、版本、时间、菜单结构、菜品 ID 和偏好互斥；
3. 显示导出时间、菜单日期、喜欢数量和不喜欢数量；
4. 用户确认后覆盖本地数据并立即更新 Pinia。

非法文件、未知版本、缺失字段或无效菜品数据会被拒绝，不改动当前状态。

如果备份菜单日期等于当前本地日期，则恢复菜单与偏好；如果日期不同，只恢复偏好，并根据当前日期与导入后的偏好重新生成菜单。

导入和导出均只在用户设备中执行，备份内容不会上传到 GitHub。

## 存储可靠性

移动端本地存储仍可能因清理网站数据、无痕模式、更换浏览器、卸载应用、仓库改名或更换域名而丢失。本次通过手动 JSON 备份降低风险，但不承诺云端级别的数据可靠性。

GitHub Pages 上的数据按设备和浏览器隔离。电脑和手机打开同一地址时会拥有各自的菜单与偏好。

## 依赖与构建

新增运行依赖：

- `pinia`；
- `vue-router`。

使用项目声明的 pnpm 安装依赖，并生成根目录 `pnpm-lock.yaml`。GitHub Actions 使用锁文件和 `--frozen-lockfile`，确保本地与线上依赖一致。

Vite 的线上 `base` 从 GitHub Actions 环境中的仓库名推导为 `/three-meals-a-day/`，本地开发使用 `/`。Hash Router 使用相同 base 创建 history。

## GitHub Pages 部署

目标仓库：`https://github.com/mu-shan/three-meals-a-day`，可见性为公开。

新增 `.github/workflows/deploy.yml`，在以下情况下执行：

- 推送到 `main`；
- 用户从 GitHub Actions 页面手动触发。

工作流步骤：

1. checkout 仓库；
2. 安装项目指定 pnpm 与 Node；
3. 使用 `pnpm install --frozen-lockfile` 安装依赖；
4. 执行 `pnpm test`；
5. 执行 `pnpm typecheck`；
6. 执行 `pnpm build`；
7. 上传 `dist` 为 Pages artifact；
8. 部署到 GitHub Pages environment。

工作流只在测试、类型检查和构建全部通过后发布。并发策略会取消同一分支尚未完成的旧部署，避免过时版本覆盖新版本。

## Git 与外部操作

当前目录不是 Git 仓库。实施阶段将：

1. 添加 Vue/Vite 项目 `.gitignore`；
2. 初始化 Git，默认分支为 `main`；
3. 提交经过验证的项目文件；
4. 使用当前 GitHub CLI 登录身份创建公开仓库 `mu-shan/three-meals-a-day`；
5. 设置 `origin` 并推送 `main`；
6. 启用 GitHub Pages 的 GitHub Actions 发布方式；
7. 等待首次部署完成并确认线上地址。

创建仓库前验证 GitHub CLI 的登录账号。如果未登录、账号不是 `mu-shan` 或权限不足，则停止外部写入并提示用户授权，不会换用其他账号。

## 错误处理

- Router 未知路径重定向首页；
- store 初始化遇到不可行偏好时恢复必要候选并提示；
- 菜单异步生成失败时结束加载状态，不覆盖当前菜单；
- localStorage 不可用时状态在当前运行期继续可用；
- 导入解析失败时显示具体原因，当前数据保持不变；
- 导入确认前不写入 store 或 localStorage；
- 下载文件失败时保留当前状态并提示用户重试；
- GitHub Actions 任何验证失败时停止部署，不发布失败产物。
