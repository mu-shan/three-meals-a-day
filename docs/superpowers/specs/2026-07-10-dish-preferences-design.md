# 菜品口味偏好设计

## 目标

在现有一日三餐随机菜单中增加可持续积累的口味偏好：用户可以直接在菜品卡上标记喜欢或不喜欢，也可以在统一的“口味偏好”菜单中管理记录。喜欢的菜提高被随机到的概率，不喜欢的菜不再进入随机候选池。

同时从菜品数据中移除“鲜肉小馄饨”，并兼容浏览器中已经保存过该菜的旧菜单。

## 偏好归属

偏好归属由菜品角色自动确定，点击时不再要求用户选择人物：

- `baby`：宝宝偏好；
- `spicy`：妈妈偏好；
- `shared`：全家偏好；
- `staple`、`drink`、`fruit`：全家偏好。

全家共享菜被标记为不喜欢时，按全家不喜欢处理。

## 数据模型与持久化

偏好数据独立于当天菜单保存，结构包含喜欢菜品 ID 和不喜欢菜品 ID 两个集合。数据写入 `localStorage`，刷新或次日重新生成菜单后继续生效。

同一道菜不能同时处于喜欢和不喜欢状态：

- 标记为喜欢时，从不喜欢集合中移除；
- 标记为不喜欢时，从喜欢集合中移除；
- 从不喜欢列表恢复时，仅恢复为普通候选，不自动标记为喜欢。

读取存储数据时进行结构校验，并忽略菜品数据中已经不存在的 ID，确保移除馄饨后旧偏好不会影响应用。

## 随机规则

所有随机入口都使用同一套偏好规则，包括首次生成、全部重抽、单餐重抽和单道换菜。

- 普通候选菜的随机权重为 `1`；
- 喜欢菜的随机权重为 `2`；
- 不喜欢菜从候选池中完全移除；
- 喜欢状态只能开关，重复点击不会继续叠加权重。

权重通过加权抽样实现，不复制或修改原始菜品数据。

点击当前菜单中的“不喜欢”后，应用立即保存偏好，并把该菜替换为同餐次、同角色且未被屏蔽的其他菜。替换仍遵守米饭午餐宝宝固定三选一等已有业务规则。

如果某个必选类别已经只剩最后一道可选菜，则拒绝继续屏蔽该菜，并提示“至少保留一道可选菜”，避免生成器进入无候选状态。

## 旧菜单兼容

应用启动并读取当天缓存菜单后，会检查每道菜是否仍存在、是否已经被标记为不喜欢。不合法的菜会按原餐次与角色进行替换；如果整餐无法安全修复，则重新生成该餐。

“鲜肉小馄饨”从菜品数据中删除后不再参与任何随机。旧缓存中存在该菜时，会在初始化检查中被替换。

## 界面与交互

页面顶部增加“口味偏好”入口，并显示已记录的偏好总数。

点击入口后：

- 桌面端从右侧打开抽屉；
- 手机端使用接近全屏的底部面板；
- 抽屉分为“喜欢的菜”和“不喜欢的菜”两个标签页；
- 每项显示菜品缩略图、名称和妈妈、宝宝或全家归属；
- 喜欢列表提供“取消喜欢”；
- 不喜欢列表提供“重新加入”；
- 空列表展示引导用户在菜品卡上标记的空状态。

菜品卡增加“喜欢”和“不喜欢”按钮。喜欢状态高亮，再次点击取消；点击不喜欢后沿用现有换菜动效立即替换。操作后提供轻量文本反馈，不使用阻塞式确认框。

抽屉支持遮罩点击、关闭按钮和 `Escape` 键关闭，并提供合适的可访问性标签。

## 模块边界

- `useDishPreferences`：管理响应式偏好状态和用户操作；
- `dishPreferenceStorage`：负责偏好的序列化、校验与持久化；
- `menuGenerator`：统一执行候选过滤、加权抽样和生成规则；
- `DishPreferenceDrawer`：展示并管理喜欢与不喜欢列表；
- `DishCard`：展示菜品偏好状态并发出操作事件；
- `App.vue`：协调菜单、偏好、即时换菜和初始化清理。

各模块只通过菜品 ID 和明确的事件交互，组件不直接操作 `localStorage`。

## 异常处理

- 存储不可用或数据损坏时，回退为空偏好，随机菜单继续工作；
- 必选类别无替换候选时，不写入不喜欢记录，并向用户说明原因；
- 已移除菜品或无效偏好 ID 在读取时自动忽略；
- 偏好修改后的存储失败不阻塞当前页面操作，但偏好仅在本次运行有效。

## 验证范围

自动化测试覆盖：

- 不喜欢菜不会被首次生成、重抽或单道换菜选中；
- 喜欢菜使用固定两倍权重，且不会累计；
- 点击不喜欢后的替换保留原有餐次和角色约束；
- 最后一个必选候选不能被屏蔽；
- 偏好能够保存、恢复并清理无效 ID；
- 旧缓存中的馄饨或不喜欢菜会被替换；
- 偏好抽屉能切换列表、取消喜欢和重新加入；
- 菜品卡能正确显示状态并发出喜欢、不喜欢事件。

验证命令为 `pnpm test` 和 `pnpm typecheck`。除非用户另行要求，不执行 `pnpm build`，也不主动打开浏览器。
