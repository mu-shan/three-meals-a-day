# “今天吃点啥”家庭三餐随机产品设计

## 1. 产品目标

为一家三口提供简单、可靠、具有家庭氛围的一日三餐随机菜单。用户打开页面后，可一键生成早餐、午餐、晚餐，也可单独更换某一餐或某一道菜。产品重点解决“每天不知道吃什么”的决策疲劳，同时保证 3 岁宝宝和妈妈的饮食需求。

第一版是无需登录和后端的单页 Web 产品，采用 Vue 3、TypeScript 和 Vite。菜单数据、随机规则与食材数据均保存在前端；当天菜单保存到浏览器本地。

## 2. 使用者与饮食约束

- 一家三口共用一桌菜，不为成员生成三套独立菜单。
- 宝宝 3 岁多，菜单需要包含宝宝友好菜，默认少盐、不辣。
- 妈妈爱吃辣；米饭午餐必须包含至少一道辣菜。
- 爸爸不设忌口，可食用整桌菜。

## 3. 产品形态

采用“今日餐桌卡片”单页方案。

- 顶部展示产品名、当天日期、家庭卡通插画和文案“今天也要好好吃饭”。
- 主操作“一键开饭”同时生成三餐。
- 页面主体展示早餐、午餐、晚餐三张卡片，午餐作为视觉重点。
- 每一餐提供“换一桌”，每道菜提供单独换菜操作。
- 页面底部展示“今日采购清单”，并随菜单实时更新。
- 移动端采用纵向卡片流；桌面端采用三餐并排布局。

## 4. 视觉设计

整体采用温暖、自然、卡通的家庭餐桌风格，以 `animal-island-vue` 为基础组件库。

- 使用其 `Button`、`Card`、`Modal`、`Time`、`Loading` 等组件。
- 使用有机圆角、立体按压按钮、柔和阴影和自然纹理。
- 主色为奶油黄、蔬菜绿和番茄红，避免通用后台式白底卡片风格。
- 菜品使用卡通食物图片；不使用任天堂官方素材。
- 菜品图片加载失败时显示本地餐盘或蔬菜卡通占位图。
- 菜品卡显示名称与角色标签：`宝宝友好`、`妈妈辣菜`、`全家共享`。
- 随机生成时使用轻量的翻牌、上菜或小岛加载动画。

组件库仓库采用 MIT 协议，但 README 同时声明仅限学习和非商业使用。当前产品按家庭自用设计；若未来用于商业用途，需要替换为自主实现的同风格组件。

## 5. 三餐随机规则

### 5.1 早餐

早餐固定生成以下组合：

1. 一份主食；
2. 一份蛋奶或豆浆；
3. 一份水果。

早餐菜品默认适合宝宝食用。例如：小米粥、蒸蛋、香蕉。

### 5.2 午餐

午餐先随机主食，包括米饭、面条、饺子、馄饨等。

当主食为米饭时，必须生成至少 3 道菜，并满足：

1. 宝宝固定菜从西红柿炒鸡蛋、土豆丝、炒豆芽中随机选择 1 道；
2. 妈妈辣菜至少 1 道，从辣椒炒肉、麻婆豆腐、香辣鸡翅等辣菜中随机；
3. 其余菜位从荤菜、素菜或汤中补足；
4. 同一桌不出现重复菜品。

当主食为面条、饺子、馄饨等一餐式主食时，搭配 1 道宝宝友好配菜和 1 道凉菜或汤，不强制妈妈辣菜。

### 5.3 晚餐

晚餐生成以下组合：

1. 一份主食；
2. 两到三道菜或汤；
3. 至少一道宝宝友好菜；
4. 不强制辣菜，整体口味比午餐清淡。

### 5.4 换菜规则

- “换一桌”重新生成整餐，但仍需满足对应餐次约束。
- 单道换菜保留原菜品角色：宝宝菜只能换成宝宝菜，妈妈辣菜只能换成另一道辣菜。
- 同一天三餐尽量避免重复菜品。
- 菜品池不足时从同角色、同餐次的可用菜品安全降级，不产生空卡片或无限循环。

## 6. 菜品与食材数据

第一版内置约 50 至 70 个常见家常菜。每个菜品包含：

- 唯一标识；
- 菜名；
- 适用餐次；
- 菜品类别；
- 口味与辣度；
- 适合成员及角色标签；
- 主要食材；
- 卡通图片地址与本地占位图。

菜品数据与随机算法分离，后续可直接扩充菜品池而无需修改页面组件。

## 7. 今日采购清单

- 根据当前三餐的食材实时生成。
- 同名食材自动去重并合并展示。
- 按蔬菜、肉蛋奶、主食、水果四类分组。
- 第一版仅展示食材名称，不计算克数和份量。
- 更换整餐或单道菜后立即重新计算。

## 8. 状态与数据流

1. 页面首次打开时读取浏览器中的当天菜单。
2. 如果存在当天菜单，直接展示；如果不存在，则生成新的三餐。
3. 用户执行一键生成、换一桌或换一道菜后，随机规则模块返回合法菜单。
4. 菜单状态更新后，采购清单由当前菜品食材重新派生。
5. 菜单与日期保存到浏览器本地；日期变化后生成新一天菜单。

浏览器存储失败时，产品仍可在当前页面会话内正常随机，只是不保留刷新后的结果。

## 9. 模块边界

- 菜品数据模块：提供结构化菜品与食材数据。
- 随机规则模块：生成三餐、校验角色约束、执行角色安全的换菜。
- 菜单状态模块：管理当前菜单、本地保存和日期切换。
- 采购清单模块：从菜单派生分类食材列表。
- 页面组件：负责展示与触发操作，不包含随机约束实现。

## 10. 异常处理

- 图片失败：使用本地占位图。
- 本地存储失败：保留内存状态，并避免阻断页面交互。
- 菜品池不足：使用同类别候选安全降级；无可用候选时保留原菜品。
- 快速连续点击：随机按钮在动画期间短暂禁用，避免状态竞争。

## 11. 验收与测试

重点验收：

- 米饭午餐始终不少于 3 道菜。
- 米饭午餐始终包含三选一宝宝固定菜。
- 米饭午餐始终包含至少一道妈妈辣菜。
- 爸爸不受角色限制。
- 单道换菜后原角色约束不丢失。
- 早餐和晚餐组合符合各自规则。
- 同一餐不出现重复菜品。
- 采购清单在所有换菜操作后同步更新。
- 刷新页面可恢复当天菜单，日期变化会生成新菜单。
- 图片失败与本地存储失败不阻断核心功能。

验证方式：

- 为随机规则、换菜约束和采购清单编写单元测试。
- 执行 TypeScript 静态检查。
- 启动开发服务检查桌面端与移动端布局和交互。
- 除非用户明确要求，不执行 `pnpm build`，不主动打开浏览器进行验证。

## 12. 第一版范围外

- 用户登录与云同步；
- 后端服务与管理后台；
- 精确克数、营养和热量计算；
- 自动下单买菜；
- 一周菜单日历；
- 多家庭或自定义家庭成员配置。
